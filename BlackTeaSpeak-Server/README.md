# BlackTeaSpeak Compat

Rust-based baseline for a future BlackTeaSpeak-compatible reimplementation.

The crate consumes the reverse-engineering foundation in `reverse-engineering/foundation` and exposes three immediate workflows:

- `cargo run -- info` prints the discovered foundation counts and binary identity.
- `cargo run -- exec "login serveradmin serveradmin"` executes a single query line against the in-memory baseline.
- `cargo run -- repl` starts a minimal interactive query shell.
- `cargo run -- serve --bind 127.0.0.1:10101` starts a TCP-based ServerQuery-compatible transport on top of the same baseline runtime.
- `cargo run -- generate-web-cert` writes a reusable localhost/127.0.0.1 certificate pair to `BlackTeaSpeak-Server/data/tls/` so the WSS transport can use a stable certificate that you can trust in Windows.
- `cargo run -- serve-web --bind 127.0.0.1:9987` starts a minimal BlackTeaWeb-oriented WSS transport plus the static web client host on `--web-client-bind` as HTTPS; by default both prefer a previously generated compat certificate under `BlackTeaSpeak-Server/data/tls/`, otherwise they first try `BlackTeaSpeak-1.5.6/certs/default_certificate.pem` plus `default_privatekey.pem`, fall back to an ephemeral localhost self-signed certificate if rustls rejects that pair, and accept overrides via `--cert` and `--key`.

Admin-key helper:

- `scripts/New-ServerAdminPrivilegeKey.ps1` connects to the running TCP query transport, resolves `serverinstance_template_serveradmin_group` automatically, runs `privilegekeyadd`, and returns the extracted token. Example: `powershell -ExecutionPolicy Bypass -File .\scripts\New-ServerAdminPrivilegeKey.ps1 -PassThruToken`.

Transport notes:

- the server sends a compact compat banner instead of claiming to be byte-identical to the original BlackTeaSpeak banner
- the BlackTeaWeb transport is intentionally narrower than the ServerQuery transport today: it currently covers `enable-raw-commands`, `handshakebegin`, `handshakeidentityproof`, `handshakeindentityproof`, `clientinit`, automatic initial `channellist` plus `notifycliententerview` plus `channellistfinished`, live shared `notifycliententerview` on BlackTeaWeb joins, `clientmove` with `notifyclientmoved`, disconnect-driven `notifyclientleftview`, `whoami`, `serverinfo`, `servergetvariables`, `serverrequestconnectioninfo`, `permissionlist`, `listfeaturesupport`, `clientupdate` with server-side nickname/avatar/away/audio-flag state plus `notifyclientupdated` for self and other BlackTeaWeb peers in the same server, `channelsubscribe`, `channelunsubscribe`, `channelsubscribeall`, `channelunsubscribeall`, and JSON ping/pong, enough for the first BlackTeaWeb login path plus the next self-state, tree, and server-info bootstrap steps
- `quit` is implemented so a TCP client can terminate its own session cleanly
- runtime state is shared across TCP clients, while authentication, selected virtual server, and notification subscriptions stay scoped to each connection
- the default BlackTeaWeb bind is `127.0.0.1:9987`; if `BlackTeaSpeak-Server/data/tls/blackteaweb-localhost-cert.pem` plus `blackteaweb-localhost-key.pem` exist, the transport prefers that stable compat certificate pair, otherwise it first tries the shipped BlackTeaSpeak certificate pair under `BlackTeaSpeak-1.5.6/certs/`, and if rustls cannot use that certificate version it falls back automatically to a generated localhost self-signed certificate so the WSS endpoint still starts cleanly
- the default runtime state file is stored at `BlackTeaSpeak-Server/data/runtime-state.json`; tests now override this path for isolated restart scenarios so suite order does not leak state across runs
- `servernotifyregister event=server` currently emits compat `notifycliententerview`, `notifyclientleftview`, `notifyclientupdated`, and `notifyserveredited` events when query clients enter, leave, change their visible client state, or edit selected virtual-server metadata on the selected virtual server
- `servernotifyregister event=channel id=<cid>` currently emits compat `notifycliententerview` and `notifyclientleftview` events when query clients join or leave that channel, plus `notifychannelcreated`, `notifychanneldeleted`, `notifychanneledited`, and `notifychannelmoved` for matching channel-tree mutations
- `servernotifyregister event=textserver`, `event=textchannel id=<cid>`, and `event=textprivate` now receive compat `notifytextmessage` events from `sendtextmessage targetmode=3|2|1`
- `servernotifyunregister` clears all session-local subscriptions immediately
- `permget` resolves the effective permission set from group assignments plus direct client overrides, while `permfind` exposes persisted server-group and direct-client assignments with stable compat synthetic permission IDs when no original numeric ID is available; both `permfind` and `permoverview` now emit deterministic rows in target order (`t=0` through `t=4`) with stable sorting inside each target scope
- `permissionlist` and `permidgetbyname` expose the shared compat permission catalog from the generated foundation, including exact IDs found in shipped command examples and pattern-derived descriptions for permissions that have no local original `permdesc` table
- `channelgroupadd`, `channelgroupaddperm`, `channelgroupcopy`, `channelgroupdel`, `channelgroupdelperm`, `channelgrouprename`, and `setclientchannelgroup` now mutate persisted compat channel-group metadata and per-channel client assignments; `channelgroupcopy` accepts both `tcgid` and the shipped doc typo `tsgid`, `channelgrouppermlist` mirrors the shipped query output more closely by emitting plain permission rows and only switching from `permid` to `permsid` when `-permsid` is requested, `channelgroupclientlist` exposes the persisted per-channel assignments filtered by `cid`, `cldbid`, and `cgid`, and `permoverview` resolves effective channel-group permissions through explicit assignments first and otherwise through the default guest channel group
- `channelclientaddperm` and `channelclientdelperm` now mutate persisted per-channel client permissions keyed by `cid` plus `cldbid`; `channelclientpermlist` mirrors the shipped query output by emitting `cid` and `cldbid` on every permission row, and `permfind` plus `permoverview` now surface these assignments as type `4` client-channel entries
- `channeladdperm` and `channeldelperm` now mutate persisted per-channel permissions stored on the channel records themselves; `channelpermlist` mirrors the shipped query output by emitting `cid` on every permission row, and `permfind` plus `permoverview` now surface these channel assignments as type `2` entries
- `servergroupadd`, `servergroupcopy`, `servergrouprename`, `servergroupautoaddperm`, `servergroupautodelperm`, and `servergroupdel` now mutate persisted compat server-group metadata; `servergroupcopy` supports overwriting an existing target group while preserving the target identity, `servergrouppermlist` now mirrors the shipped query output more closely by emitting plain permission rows and only switching from `permid` to `permsid` when `-permsid` is requested, the auto-permission commands target groups through `i_group_auto_update_type`, `privilegekeyadd` and `privilegekeydelete` are direct aliases to `tokenadd` and `tokendelete`, and `tokenadd`, `tokendelete`, `tokenedit`, `tokenactionlist`, `tokenlist`, `tokenuse`, `privilegekeylist`, and `privilegekeyuse` operate on persisted compat token metadata with stored action lists, while `tokenuse` currently applies supported server-group grant actions, enforces expiry and max-uses, and rejects unsupported action types explicitly

This is a development base, not a drop-in server replacement. Broader protocol fidelity, deeper permissions enforcement, richer persistence coverage, and most commands are intentionally still open work.
